# Requirements Document

## Introduction

Система Message Template Configurator предоставляет удобный CLI-интерфейс для управления шаблонами push-уведомлений. Инструмент позволяет администратору просматривать, редактировать, создавать и валидировать шаблоны сообщений для всех каналов (Telegram, WhatsApp, MAX) и типов уведомлений без необходимости прямого доступа к базе данных. Конфигуратор включает предпросмотр сообщений с подстановкой тестовых данных, сброс шаблонов к значениям по умолчанию и управление переменными шаблонов.

## Glossary

- **Template_Configurator**: CLI-интерфейс управления шаблонами сообщений, реализованный как Python-скрипт с интерактивным меню
- **Template_Record**: Запись шаблона в базе данных (модель NotificationTemplate), содержащая тип уведомления, канал, язык, тело шаблона и метаданные
- **Template_Variable**: Именованная переменная-плейсхолдер в формате `{{variable_name}}`, подставляемая при рендеринге шаблона (допустимые: client_name, master_name, service_name, appointment_date, appointment_time, salon_address, salon_phone, salon_name)
- **Template_Preview**: Результат рендеринга шаблона с подстановкой тестовых данных для визуальной проверки корректности
- **Template_Validator**: Компонент, проверяющий синтаксис Jinja2 и допустимость используемых переменных в шаблоне
- **Channel**: Канал доставки уведомлений (telegram, whatsapp, max)
- **Notification_Type**: Тип уведомления (new_appointment, confirmed, cancelled, in_progress, changed, reminder_24h, reminder_2h, birthday)
- **Default_Template**: Шаблон, поставляемый с системой и используемый при отсутствии пользовательского шаблона (is_default=True)
- **Sample_Data**: Набор тестовых значений переменных для предпросмотра шаблонов

## Requirements

### Requirement 1: Запуск Template Configurator

**User Story:** Как администратор сервиса, я хочу запускать конфигуратор шаблонов одной командой, чтобы быстро перейти к управлению шаблонами уведомлений.

#### Acceptance Criteria

1. WHEN администратор выполняет команду `python -m app.template_config`, THE Template_Configurator SHALL запустить интерактивное меню управления шаблонами в терминале в течение 5 секунд
2. WHEN Template_Configurator запускается, THE Template_Configurator SHALL установить соединение с PostgreSQL и отобразить главное меню со следующими пунктами: Просмотр шаблонов, Редактирование шаблона, Создание шаблона, Предпросмотр шаблона, Сброс к значениям по умолчанию, Управление переменными, Выход
3. IF Template_Configurator не может установить соединение с базой данных в течение 10 секунд, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке с указанием причины и завершить работу с кодом возврата 1
4. WHEN Template_Configurator запускается, THE Template_Configurator SHALL отобразить краткую статистику: общее количество шаблонов, количество активных шаблонов, количество пользовательских шаблонов (is_default=False)
5. IF Template_Configurator запущен в неинтерактивном окружении (отсутствует TTY), THEN THE Template_Configurator SHALL завершить работу с кодом возврата 1 и сообщением о необходимости интерактивного терминала

### Requirement 2: Просмотр шаблонов

**User Story:** Как администратор, я хочу просматривать все существующие шаблоны с фильтрацией по каналу и типу, чтобы быстро находить нужный шаблон.

#### Acceptance Criteria

1. WHEN администратор выбирает пункт "Просмотр шаблонов", THE Template_Configurator SHALL отобразить список фильтров: все шаблоны, по каналу (telegram, whatsapp, max), по типу уведомления (new_appointment, confirmed, cancelled, in_progress, changed, reminder_24h, reminder_2h, birthday), с возможностью одновременного применения фильтра по каналу и фильтра по типу уведомления
2. WHEN администратор выбирает фильтр, THE Template_Configurator SHALL в течение не более 3 секунд отобразить таблицу шаблонов, содержащую столбцы: ID, тип уведомления, канал, язык, статус (активен/неактивен), источник (system — если поле created_by равно NULL или "system", custom — в остальных случаях), дата последнего изменения, отсортированную по дате последнего изменения от новых к старым
3. WHEN администратор выбирает конкретный шаблон из таблицы по его ID, THE Template_Configurator SHALL отобразить полное содержимое шаблона: тело шаблона, в котором переменные в формате `{{имя_переменной}}` визуально выделены относительно окружающего текста, список всех используемых переменных, извлечённых из тела шаблона, и метаданные (создатель, дата создания, дата последнего изменения)
4. WHILE список шаблонов содержит более 10 записей, THE Template_Configurator SHALL отображать результаты постранично по 10 записей с навигацией вперёд/назад и отображением текущей страницы и общего количества страниц в формате "Страница N из M"
5. IF запрошенный фильтр не возвращает ни одного шаблона, THEN THE Template_Configurator SHALL отобразить сообщение об отсутствии шаблонов для выбранного фильтра и предложить создать новый шаблон
6. IF при загрузке списка шаблонов происходит ошибка соединения с базой данных или превышен таймаут в 3 секунды, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке с указанием причины и предложить повторить запрос

### Requirement 3: Редактирование шаблона

**User Story:** Как администратор, я хочу редактировать тело существующего шаблона с немедленной валидацией, чтобы быстро обновлять тексты уведомлений.

#### Acceptance Criteria

1. WHEN администратор выбирает пункт "Редактирование шаблона", THE Template_Configurator SHALL запросить выбор шаблона по каналу (telegram, whatsapp, max) и типу уведомления (new_appointment, confirmed, cancelled, in_progress, changed, reminder_24h, reminder_2h, birthday) или по числовому ID шаблона
2. IF указанный шаблон не найден в базе данных (несуществующий ID или комбинация канала и типа), THEN THE Template_Configurator SHALL отобразить сообщение об ошибке с указанием что шаблон не найден и вернуться к запросу выбора шаблона
3. WHEN шаблон выбран для редактирования, THE Template_Configurator SHALL отобразить текущее тело шаблона и предложить ввести новый текст в многострочном режиме (завершение ввода по пустой строке или комбинации Ctrl+D), при этом длина нового текста не должна превышать 4096 символов
4. WHEN администратор вводит новый текст шаблона, THE Template_Validator SHALL проверить синтаксис Jinja2 и допустимость всех используемых переменных (допустимые: client_name, master_name, service_name, appointment_date, appointment_time, salon_address, salon_phone, salon_name) до сохранения
5. IF Template_Validator обнаруживает недопустимые переменные, THEN THE Template_Configurator SHALL отобразить список недопустимых переменных, список допустимых переменных и запросить исправленный текст шаблона, допуская не более 5 последовательных неудачных попыток валидации, после чего предложить отменить редактирование
6. IF Template_Validator обнаруживает синтаксическую ошибку Jinja2, THEN THE Template_Configurator SHALL отобразить описание ошибки с указанием позиции и запросить исправленный текст шаблона, допуская не более 5 последовательных неудачных попыток валидации, после чего предложить отменить редактирование
7. WHEN валидация пройдена успешно, THE Template_Configurator SHALL отобразить предпросмотр отрендеренного шаблона с Sample_Data и запросить подтверждение сохранения
8. WHEN администратор подтверждает сохранение, THE Template_Configurator SHALL обновить Template_Record в базе данных и инвалидировать соответствующий ключ в Redis-кеше в формате template:{notification_type}:{channel}:{language}
9. WHEN администратор отклоняет сохранение, THE Template_Configurator SHALL вернуться к редактированию без изменения Template_Record
10. IF шаблон является Default_Template (is_default=True), THEN THE Template_Configurator SHALL предупредить администратора о редактировании системного шаблона и запросить явное подтверждение перед началом редактирования

### Requirement 4: Создание нового шаблона

**User Story:** Как администратор, я хочу создавать новые шаблоны для конкретных комбинаций канала и типа уведомления, чтобы расширять набор доступных шаблонов.

#### Acceptance Criteria

1. WHEN администратор выбирает пункт "Создание шаблона", THE Template_Configurator SHALL последовательно запросить: канал (telegram, whatsapp, max), тип уведомления (new_appointment, confirmed, cancelled, in_progress, changed, reminder_24h, reminder_2h, birthday), язык (по умолчанию "ru")
2. IF для выбранной комбинации канала, типа и языка уже существует активный Template_Record, THEN THE Template_Configurator SHALL отобразить предупреждение о существующем шаблоне и предложить выбор: заменить существующий, создать неактивную копию или отменить создание
3. WHEN параметры шаблона определены, THE Template_Configurator SHALL предложить выбор: ввести текст с нуля, использовать существующий активный шаблон того же типа из другого канала как основу (если такой шаблон существует), использовать Default_Template как основу
4. IF администратор выбирает вариант "использовать шаблон из другого канала" и ни одного активного шаблона того же типа в других каналах не существует, THEN THE Template_Configurator SHALL отобразить сообщение об отсутствии подходящих шаблонов и вернуться к выбору способа создания
5. WHEN администратор вводит тело шаблона, THE Template_Validator SHALL выполнить валидацию синтаксиса Jinja2 и проверку допустимости переменных (допустимые: client_name, master_name, service_name, appointment_date, appointment_time, salon_address, salon_phone, salon_name) до сохранения
6. IF валидация тела шаблона выявляет ошибки, THEN THE Template_Validator SHALL отобразить список обнаруженных ошибок и предложить администратору исправить текст шаблона или отменить создание
7. WHEN валидация пройдена, THE Template_Configurator SHALL отобразить предпросмотр шаблона, отрендеренного с Sample_Data, и запросить подтверждение сохранения
8. WHEN администратор подтверждает создание, THE Template_Configurator SHALL сохранить новый Template_Record в базе данных с полем created_by="template_configurator" и body_template длиной не более 4096 символов
9. IF сохранение в базу данных завершается ошибкой, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке с указанием причины и предложить повторить попытку сохранения или отменить создание

### Requirement 5: Предпросмотр шаблона

**User Story:** Как администратор, я хочу видеть, как будет выглядеть сообщение с реальными данными, чтобы убедиться в корректности форматирования перед сохранением.

#### Acceptance Criteria

1. WHEN администратор выбирает пункт "Предпросмотр шаблона", THE Template_Configurator SHALL отобразить список доступных шаблонов, сгруппированных по каналу (telegram, max, whatsapp) и типу уведомления, и запросить выбор шаблона по номеру из списка или по ID
2. WHEN шаблон выбран для предпросмотра, THE Template_Configurator SHALL отрендерить шаблон с набором Sample_Data, содержащим реалистичные значения: client_name="Анна Иванова", master_name="Елена Петрова", service_name="Стрижка и укладка", appointment_date="15 января 2025", appointment_time="14:30", salon_address="ул. Пушкина, д. 10", salon_phone="+7 (999) 123-45-67", salon_name="Салон Красоты", и отобразить результат в течение не более 5 секунд
3. IF выбранный шаблон не найден или неактивен, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке и вернуться к списку доступных шаблонов
4. IF рендеринг шаблона завершается ошибкой синтаксиса Jinja2, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке с указанием позиции синтаксической ошибки и предложить вернуться к списку шаблонов
5. WHEN предпросмотр отображён, THE Template_Configurator SHALL предложить администратору ввести собственные значения переменных для повторного предпросмотра, отображая список всех переменных шаблона с текущими значениями из Sample_Data
6. WHEN администратор вводит пользовательские значения переменных, THE Template_Configurator SHALL отрендерить шаблон с указанными значениями, подставляя Sample_Data для незаполненных переменных
7. WHEN канал шаблона — telegram, THE Template_Configurator SHALL отобразить перед результатом предпросмотра аннотацию о поддержке HTML-разметки (bold, italic, code)
8. WHEN канал шаблона — max, THE Template_Configurator SHALL отобразить перед результатом предпросмотра аннотацию о поддержке Markdown-разметки (bold, italic)
9. WHEN канал шаблона — whatsapp, THE Template_Configurator SHALL отобразить перед результатом предпросмотра аннотацию об ограничении: только plain text без форматирования
10. WHEN администратор вводит команду "exit" или "back" в режиме предпросмотра, THE Template_Configurator SHALL завершить режим предпросмотра и вернуться к предыдущему меню

### Requirement 6: Сброс шаблонов к значениям по умолчанию

**User Story:** Как администратор, я хочу сбросить изменённые шаблоны к исходным значениям, чтобы восстановить работоспособность при ошибках в пользовательских шаблонах.

#### Acceptance Criteria

1. WHEN администратор выбирает пункт "Сброс к значениям по умолчанию", THE Template_Configurator SHALL предложить выбор: сбросить конкретный шаблон, сбросить все шаблоны канала, сбросить все шаблоны
2. WHEN администратор выбирает сброс конкретного шаблона, THE Template_Configurator SHALL отобразить текущее значение body_template и соответствующее значение из DEFAULT_TEMPLATES в двух смежных областях для визуального сравнения
3. WHEN администратор подтверждает сброс конкретного шаблона, THE Template_Configurator SHALL заменить body_template выбранного Template_Record на значение из DEFAULT_TEMPLATES, установить is_default=True, обновить поле created_by значением "system" и инвалидировать Redis-кеш для соответствующей комбинации notification_type, channel и language
4. IF для выбранной комбинации канала и типа отсутствует значение в DEFAULT_TEMPLATES, THEN THE Template_Configurator SHALL отобразить сообщение об отсутствии шаблона по умолчанию и предложить удалить текущий шаблон или отменить операцию
5. WHEN администратор выбирает массовый сброс (по каналу или все), THE Template_Configurator SHALL отобразить количество шаблонов, которые будут затронуты, и запросить подтверждение вводом слова "RESET" для предотвращения случайного сброса
6. WHEN массовый сброс подтверждён, THE Template_Configurator SHALL последовательно сбросить каждый шаблон (заменить body_template, установить is_default=True, обновить created_by="system", инвалидировать Redis-кеш), отображая прогресс в формате "Обработано N из M шаблонов"
7. IF в процессе массового сброса происходит ошибка при обработке отдельного шаблона, THEN THE Template_Configurator SHALL пропустить этот шаблон, продолжить обработку оставшихся и по завершении отобразить итоговый отчёт с количеством успешно сброшенных шаблонов и списком шаблонов, для которых сброс не удался

### Requirement 7: Управление переменными шаблонов

**User Story:** Как администратор, я хочу видеть список доступных переменных с описанием и примерами, чтобы корректно использовать их при создании шаблонов.

#### Acceptance Criteria

1. WHEN администратор выбирает пункт "Управление переменными", THE Template_Configurator SHALL в течение не более 3 секунд отобразить таблицу всех допустимых Template_Variable (8 переменных) с колонками: имя переменной, описание, пример значения
2. THE Template_Configurator SHALL отображать следующие описания и примеры переменных: client_name — описание "Имя клиента", пример "Анна Иванова"; master_name — описание "Имя мастера", пример "Елена Смирнова"; service_name — описание "Название услуги", пример "Стрижка женская"; appointment_date — описание "Дата записи", пример "15.03.2025"; appointment_time — описание "Время записи", пример "14:30"; salon_address — описание "Адрес салона", пример "ул. Пушкина, д. 10"; salon_phone — описание "Телефон салона", пример "+7 (495) 123-45-67"; salon_name — описание "Название салона", пример "Студия красоты Элегант"
3. WHEN администратор запрашивает информацию о конкретной переменной, THE Template_Configurator SHALL отобразить: описание переменной, пример значения, список шаблонов, использующих данную переменную, с указанием канала и типа уведомления, либо сообщение об отсутствии использования
4. WHEN администратор запрашивает анализ использования переменных, THE Template_Configurator SHALL отобразить статистику: количество активных шаблонов (is_active=true), использующих каждую переменную, и список переменных, не используемых ни в одном активном шаблоне
5. IF Template_Configurator не может загрузить данные о шаблонах из базы данных, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке и предложить администратору повторить попытку

### Requirement 8: Валидация шаблонов

**User Story:** Как администратор, я хочу проверить все шаблоны на корректность одной командой, чтобы выявить проблемы до отправки уведомлений клиентам.

#### Acceptance Criteria

1. WHEN Template_Configurator выполняет валидацию шаблона, THE Template_Validator SHALL проверить: корректность синтаксиса Jinja2, отсутствие неизвестных переменных (не входящих в ALLOWED_VARIABLES), непустое тело шаблона (длина body_template от 1 до 10000 символов)
2. WHEN администратор запрашивает массовую валидацию всех шаблонов, THE Template_Configurator SHALL проверить каждый активный Template_Record в течение не более 60 секунд суммарно и отобразить отчёт: количество валидных шаблонов, количество шаблонов с ошибками, список шаблонов с ошибками (ID, канал, тип, описание ошибки)
3. IF массовая валидация обнаруживает шаблоны с ошибками, THEN THE Template_Configurator SHALL предложить администратору перейти к редактированию первого ошибочного шаблона
4. WHEN Template_Validator проверяет шаблон, THE Template_Validator SHALL дополнительно проверить возможность рендеринга шаблона с Sample_Data в течение не более 5 секунд, и при возникновении исключения зафиксировать тип исключения и сообщение как ошибку валидации
5. IF массовая валидация не обнаруживает ни одного активного Template_Record, THEN THE Template_Configurator SHALL отобразить сообщение об отсутствии активных шаблонов для проверки и предложить вернуться в главное меню
6. IF валидация отдельного шаблона завершается непредвиденной ошибкой во время массовой проверки, THEN THE Template_Configurator SHALL зафиксировать ошибку для данного шаблона в отчёте и продолжить валидацию оставшихся шаблонов

### Requirement 9: Аргументы командной строки

**User Story:** Как администратор, я хочу выполнять частые операции без навигации по меню, чтобы ускорить работу с шаблонами.

#### Acceptance Criteria

1. WHEN администратор выполняет команду с аргументом `--list`, THE Template_Configurator SHALL вывести все шаблоны в табличном формате (столбцы: ID, notification_type, channel, language, is_active) без входа в интерактивное меню и завершить работу с кодом возврата 0
2. WHEN администратор выполняет команду с аргументами `--list --channel <channel>`, где channel — одно из допустимых значений (telegram, whatsapp, max), THE Template_Configurator SHALL вывести только шаблоны указанного канала в том же табличном формате и завершить работу с кодом возврата 0
3. WHEN администратор выполняет команду с аргументом `--preview <id>`, где id соответствует существующему шаблону, THE Template_Configurator SHALL отобразить полную информацию о шаблоне и предпросмотр с Sample_Data без входа в интерактивное меню и завершить работу с кодом возврата 0
4. WHEN администратор выполняет команду с аргументом `--validate`, THE Template_Configurator SHALL выполнить валидацию всех активных шаблонов и вывести результат для каждого шаблона (ID, статус: OK или FAIL с описанием ошибки), завершив работу с кодом возврата 0 при отсутствии ошибок или с кодом возврата 1 при наличии хотя бы одной ошибки валидации
5. WHEN администратор выполняет команду с аргументом `--reset <id>`, THE Template_Configurator SHALL запросить подтверждение (y/n) и при получении подтверждения сбросить шаблон к значению по умолчанию
6. IF указан неизвестный аргумент командной строки, THEN THE Template_Configurator SHALL отобразить справку по доступным аргументам и завершить работу с кодом возврата 1
7. IF аргумент `--preview <id>` или `--reset <id>` содержит id, не соответствующий ни одному существующему шаблону, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке и завершить работу с кодом возврата 1
8. IF аргумент `--list --channel <channel>` содержит значение channel, не входящее в список допустимых (telegram, whatsapp, max), THEN THE Template_Configurator SHALL отобразить сообщение об ошибке со списком допустимых значений каналов и завершить работу с кодом возврата 1
9. IF аргумент `--reset <id>` указывает на шаблон, для которого отсутствует значение по умолчанию, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке и завершить работу с кодом возврата 1

### Requirement 10: Обработка ошибок и прерываний

**User Story:** Как администратор, я хочу безопасно прервать работу конфигуратора без повреждения данных, чтобы не нарушить работу системы уведомлений.

#### Acceptance Criteria

1. WHEN администратор нажимает Ctrl+C во время работы Template_Configurator, THE Template_Configurator SHALL перехватить сигнал SIGINT, откатить текущую незавершённую транзакцию базы данных без сохранения промежуточных изменений и закрыть соединения с базой данных и Redis в течение не более 5 секунд
2. IF возникает ошибка соединения с базой данных во время операции, THEN THE Template_Configurator SHALL отобразить сообщение об ошибке с указанием причины и предложить повторить операцию (не более 3 попыток) или вернуться в главное меню
3. IF возникает ошибка соединения с Redis при инвалидации кеша, THEN THE Template_Configurator SHALL отобразить предупреждение о необходимости ручной инвалидации кеша (ожидание истечения TTL 5 минут) и продолжить работу, считая операцию над шаблоном успешной
4. WHEN Template_Configurator завершает работу (штатно или по ошибке), THE Template_Configurator SHALL закрыть все соединения с базой данных и Redis в течение не более 5 секунд, после чего завершить процесс с кодом возврата 0 при штатном завершении или 1 при завершении по ошибке
5. IF возникает непредвиденная ошибка во время работы Template_Configurator, THEN THE Template_Configurator SHALL откатить текущую незавершённую транзакцию, отобразить сообщение с типом ошибки и описанием, и предложить вернуться в главное меню или завершить работу
