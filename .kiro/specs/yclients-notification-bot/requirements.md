# Requirements Document

## Introduction

Система представляет собой production-ready бот-уведомитель, заменяющий платный сервис пуш-уведомлений YClients. Система интегрируется с CRM YClients через официальное API, получает события (записи, изменения статусов, напоминания, дни рождения) и доставляет персонализированные уведомления клиентам через три канала: Telegram, WhatsApp и MAX (мессенджер VK). Клиенты могут интерактивно подтверждать, отменять или переносить записи прямо из мессенджера, а изменения автоматически синхронизируются обратно в YClients.

---

## Glossary

- **System** — вся система бот-уведомителя в целом
- **Scheduler** — компонент планировщика задач (APScheduler/Celery), отвечающий за периодический опрос YClients и отправку напоминаний
- **Dispatcher** — компонент маршрутизации уведомлений, выбирающий канал доставки для каждого клиента
- **TelegramAdapter** — адаптер интеграции с Telegram Bot API
- **WhatsAppAdapter** — адаптер интеграции с WhatsApp Business Cloud API
- **MaxAdapter** — адаптер интеграции с MAX Bot API (platform-api.max.ru)
- **YClientsClient** — компонент взаимодействия с YClients API
- **WebhookHandler** — компонент обработки входящих webhook-запросов от YClients
- **NotificationService** — сервис формирования и отправки уведомлений
- **TemplateEngine** — движок шаблонов сообщений с поддержкой персонализации
- **ClientRegistry** — реестр соответствия client_id YClients → идентификаторам мессенджеров
- **RetryQueue** — очередь повторных попыток доставки сообщений
- **AdminPanel** — административный интерфейс управления системой
- **AuditLog** — журнал истории отправок и действий пользователей
- **RateLimiter** — компонент ограничения частоты запросов к внешним API
- **client_id** — уникальный идентификатор клиента в системе YClients
- **telegram_id** — уникальный идентификатор пользователя в Telegram
- **whatsapp_phone** — номер телефона клиента в формате E.164 для WhatsApp
- **max_user_id** — уникальный идентификатор пользователя в мессенджере MAX
- **Запись** — визит клиента в салон/организацию, зафиксированный в YClients
- **Шаблон** — текстовый шаблон сообщения с переменными персонализации
- **Канал** — мессенджер для доставки уведомлений (Telegram, WhatsApp, MAX)

## Requirements

### Requirement 1: YClients API Integration

**User Story:** Как администратор салона, я хочу, чтобы система автоматически получала данные о записях из YClients, чтобы клиенты своевременно получали уведомления без ручного труда.

#### Acceptance Criteria

1. THE YClientsClient SHALL подключаться к YClients API с использованием токена авторизации, указанного в переменной окружения `YCLIENTS_API_TOKEN`.
2. WHEN Scheduler выполняет плановый опрос, THE YClientsClient SHALL запрашивать список новых и изменённых записей за последние 3 минуты с интервалом от 1 до 3 минут.
3. WHEN YClients отправляет webhook-запрос на эндпоинт `/webhook/yclients`, THE WebhookHandler SHALL обработать событие в течение 5 секунд и поставить задачу в очередь.
4. IF webhook от YClients недоступен или не настроен, THEN THE Scheduler SHALL использовать polling как основной механизм получения событий.
5. THE YClientsClient SHALL поддерживать получение следующих типов событий: создание записи, изменение статуса записи, изменение мастера, изменение времени записи, день рождения клиента.
6. WHEN YClients API возвращает HTTP-статус 429 (Too Many Requests), THE YClientsClient SHALL применить экспоненциальную задержку с начальным интервалом 1 секунда и максимальным интервалом 60 секунд.
7. IF YClients API возвращает HTTP-статус 5xx три раза подряд, THEN THE YClientsClient SHALL записать ошибку в лог и прекратить попытки на 5 минут.
8. THE YClientsClient SHALL хранить токен авторизации исключительно в переменной окружения и не записывать его в логи или базу данных в открытом виде.

### Requirement 2: Реестр клиентов и привязка каналов

**User Story:** Как клиент салона, я хочу привязать свой аккаунт в мессенджере к записи в YClients, чтобы получать уведомления в удобном для меня канале.

#### Acceptance Criteria

1. THE ClientRegistry SHALL хранить для каждого client_id из YClients следующие поля: `telegram_id`, `whatsapp_phone`, `max_user_id`, `preferred_channel`, `notification_settings`, `created_at`, `updated_at`.
2. WHEN клиент впервые пишет боту в Telegram и указывает свой номер телефона, THE TelegramAdapter SHALL сопоставить номер телефона с client_id в YClients и сохранить привязку в ClientRegistry.
3. WHEN клиент впервые пишет боту в MAX и указывает свой номер телефона, THE MaxAdapter SHALL сопоставить номер телефона с client_id в YClients и сохранить привязку в ClientRegistry.
4. THE ClientRegistry SHALL поддерживать одновременную привязку нескольких каналов для одного client_id.
5. WHEN клиент отправляет команду `/unsubscribe` в любом из мессенджеров, THE System SHALL удалить привязку соответствующего канала для данного client_id из ClientRegistry.
6. IF для client_id не найдена привязка ни к одному каналу, THEN THE NotificationService SHALL записать предупреждение в лог и пропустить отправку уведомления.
7. THE ClientRegistry SHALL хранить настройки уведомлений по каждому каналу отдельно: включён/выключен, типы уведомлений, временное окно (например, не беспокоить с 22:00 до 09:00).

### Requirement 3: Отправка уведомлений через Telegram

**User Story:** Как клиент, я хочу получать красивые уведомления о своих записях в Telegram с возможностью интерактивного ответа, чтобы быстро подтверждать или отменять визиты.

#### Acceptance Criteria

1. WHEN NotificationService формирует уведомление для Telegram-канала, THE TelegramAdapter SHALL отправить сообщение с использованием Telegram Bot API метода `sendMessage` с поддержкой HTML-разметки.
2. THE TelegramAdapter SHALL прикреплять к уведомлениям о записи inline-клавиатуру с кнопками: «✅ Подтвердить», «❌ Отменить», «🔄 Перенести».
3. WHEN клиент нажимает inline-кнопку в Telegram, THE TelegramAdapter SHALL обработать callback_query в течение 3 секунд и отправить подтверждение действия.
4. THE TelegramAdapter SHALL поддерживать отправку сообщений в форматах HTML и Markdown v2.
5. WHEN Telegram API возвращает ошибку 429 (Too Many Requests), THE TelegramAdapter SHALL соблюдать значение параметра `retry_after` из ответа API перед повторной попыткой.
6. IF telegram_id клиента заблокировал бота, THEN THE TelegramAdapter SHALL получить ошибку `403 Forbidden`, записать её в AuditLog и деактивировать Telegram-канал для данного client_id в ClientRegistry.
7. THE TelegramAdapter SHALL соблюдать ограничение Telegram API: не более 30 сообщений в секунду на бота и не более 1 сообщения в секунду в один чат.

### Requirement 4: Отправка уведомлений через WhatsApp

**User Story:** Как клиент, я хочу получать уведомления о записях в WhatsApp, чтобы не пропускать важные напоминания в привычном мессенджере.

#### Acceptance Criteria

1. WHEN NotificationService формирует уведомление для WhatsApp-канала, THE WhatsAppAdapter SHALL отправить сообщение через WhatsApp Business Cloud API (graph.facebook.com/v18.0) с использованием токена из переменной окружения `WHATSAPP_API_TOKEN`.
2. THE WhatsAppAdapter SHALL использовать предварительно одобренные шаблоны сообщений (template messages) для первичного контакта с клиентом согласно требованиям WhatsApp Business Policy.
3. WHEN клиент отвечает на сообщение в WhatsApp в течение 24 часов, THE WhatsAppAdapter SHALL использовать свободный текст (free-form message) для последующих сообщений в рамках открытой сессии.
4. THE WhatsAppAdapter SHALL поддерживать отправку интерактивных сообщений типа `interactive/button` с кнопками «Подтвердить», «Отменить», «Перенести» (максимум 3 кнопки согласно ограничениям API).
5. WHEN WhatsApp API возвращает ошибку доставки, THE WhatsAppAdapter SHALL записать код ошибки и описание в AuditLog.
6. IF номер whatsapp_phone не зарегистрирован в WhatsApp, THEN THE WhatsAppAdapter SHALL получить ошибку от API, записать её в AuditLog и деактивировать WhatsApp-канал для данного client_id в ClientRegistry.
7. THE WhatsAppAdapter SHALL хранить `WHATSAPP_PHONE_NUMBER_ID` и `WHATSAPP_BUSINESS_ACCOUNT_ID` в переменных окружения.

### Requirement 5: Отправка уведомлений через MAX

**User Story:** Как клиент, использующий российский мессенджер MAX, я хочу получать уведомления о записях в MAX, чтобы не зависеть от иностранных мессенджеров.

#### Acceptance Criteria

1. WHEN NotificationService формирует уведомление для MAX-канала, THE MaxAdapter SHALL отправить сообщение через MAX Bot API (platform-api.max.ru) с использованием токена из переменной окружения `MAX_BOT_TOKEN`.
2. THE MaxAdapter SHALL поддерживать отправку сообщений с inline-клавиатурой (Keyboard) с кнопками типа `callback` для интерактивных действий.
3. WHEN клиент нажимает кнопку в MAX, THE MaxAdapter SHALL обработать callback-событие и выполнить соответствующее действие в течение 5 секунд.
4. THE MaxAdapter SHALL получать `max_user_id` клиента при первом взаимодействии клиента с ботом и сохранять его в ClientRegistry.
5. THE MaxAdapter SHALL поддерживать получение обновлений через long-polling или webhook согласно документации MAX Bot API.
6. IF MAX Bot API недоступен более 60 секунд, THEN THE MaxAdapter SHALL записать ошибку в лог и повторить попытку подключения через 30 секунд.
7. THE MaxAdapter SHALL поддерживать форматирование текста сообщений согласно возможностям MAX Bot API (жирный, курсив, ссылки).

### Requirement 6: Типы уведомлений и триггеры

**User Story:** Как администратор салона, я хочу, чтобы клиенты автоматически получали уведомления обо всех важных событиях, связанных с их записями, чтобы снизить количество неявок и отмен.

#### Acceptance Criteria

1. WHEN в YClients создаётся новая запись для клиента, THE NotificationService SHALL отправить уведомление «Новая запись» в течение 2 минут после события.
2. WHEN статус записи в YClients изменяется на «Подтверждена», THE NotificationService SHALL отправить уведомление «Запись подтверждена» в течение 2 минут.
3. WHEN статус записи в YClients изменяется на «Отменена», THE NotificationService SHALL отправить уведомление «Запись отменена» в течение 2 минут.
4. WHEN статус записи в YClients изменяется на «В работе», THE NotificationService SHALL отправить уведомление «Мастер ждёт вас» в течение 2 минут.
5. WHEN мастер или время записи изменяются в YClients, THE NotificationService SHALL отправить уведомление «Изменение записи» с указанием новых данных в течение 2 минут.
6. WHEN до времени записи остаётся 24 часа (±5 минут), THE Scheduler SHALL инициировать отправку напоминания «Напоминание за 24 часа».
7. WHEN до времени записи остаётся 2 часа (±5 минут), THE Scheduler SHALL инициировать отправку напоминания «Напоминание за 2 часа».
8. WHEN наступает день рождения клиента согласно данным YClients, THE Scheduler SHALL инициировать отправку поздравления в 10:00 по часовому поясу организации.
9. THE NotificationService SHALL не отправлять дублирующие уведомления одного типа для одной записи в течение 10 минут.
10. WHILE настройка «не беспокоить» активна для клиента, THE NotificationService SHALL откладывать отправку уведомлений до начала разрешённого временного окна, за исключением уведомлений «В работе».

### Requirement 7: Интерактивные действия клиента

**User Story:** Как клиент, я хочу подтверждать, отменять или переносить запись прямо из мессенджера, чтобы не звонить в салон.

#### Acceptance Criteria

1. WHEN клиент нажимает кнопку «✅ Подтвердить» в любом мессенджере, THE System SHALL обновить статус записи в YClients на «Подтверждена» через YClients API в течение 10 секунд.
2. WHEN клиент нажимает кнопку «❌ Отменить» в любом мессенджере, THE System SHALL запросить у клиента подтверждение отмены и, после подтверждения, обновить статус записи в YClients на «Отменена» в течение 10 секунд.
3. WHEN клиент нажимает кнопку «🔄 Перенести» в любом мессенджере, THE System SHALL отправить клиенту сообщение с инструкцией позвонить в салон или ссылкой на онлайн-запись YClients.
4. WHEN действие клиента успешно выполнено в YClients, THE System SHALL отправить клиенту подтверждающее сообщение с результатом действия.
5. IF YClients API возвращает ошибку при обновлении статуса, THEN THE System SHALL уведомить клиента об ошибке и предложить позвонить в салон, а также записать ошибку в AuditLog.
6. THE System SHALL игнорировать повторные нажатия одной и той же кнопки для одной записи в течение 30 секунд (защита от дублирования).
7. WHEN клиент отвечает текстом на уведомление в Telegram или MAX, THE System SHALL отправить клиенту справочное сообщение с доступными командами.

### Requirement 8: Движок шаблонов сообщений

**User Story:** Как администратор, я хочу настраивать шаблоны уведомлений с персонализацией, чтобы сообщения выглядели профессионально и содержали всю нужную информацию.

#### Acceptance Criteria

1. THE TemplateEngine SHALL поддерживать переменные персонализации: `{{client_name}}`, `{{master_name}}`, `{{service_name}}`, `{{appointment_date}}`, `{{appointment_time}}`, `{{salon_address}}`, `{{salon_phone}}`, `{{salon_name}}`.
2. WHEN TemplateEngine рендерит шаблон, THE TemplateEngine SHALL заменить все переменные персонализации соответствующими данными из YClients в течение 100 миллисекунд.
3. IF переменная персонализации отсутствует в данных YClients, THEN THE TemplateEngine SHALL заменить её пустой строкой и записать предупреждение в лог.
4. THE TemplateEngine SHALL поддерживать отдельные шаблоны для каждого типа уведомления и каждого канала (Telegram, WhatsApp, MAX).
5. THE TemplateEngine SHALL поддерживать использование эмодзи в шаблонах.
6. THE AdminPanel SHALL предоставлять интерфейс для редактирования шаблонов с предварительным просмотром результата.
7. WHEN администратор сохраняет новый шаблон, THE TemplateEngine SHALL валидировать корректность всех переменных персонализации и вернуть список некорректных переменных, если они есть.
8. THE System SHALL поставлять набор шаблонов по умолчанию для всех типов уведомлений на русском языке при первом запуске.

### Requirement 9: Очередь задач и механизм повторных попыток

**User Story:** Как администратор, я хочу, чтобы система надёжно доставляла уведомления даже при временных сбоях внешних API, чтобы клиенты не пропускали важные сообщения.

#### Acceptance Criteria

1. THE System SHALL использовать очередь задач (Celery с Redis или RabbitMQ в качестве брокера) для асинхронной обработки всех задач отправки уведомлений.
2. WHEN отправка уведомления завершается ошибкой, THE RetryQueue SHALL повторить попытку с экспоненциальной задержкой: 1 минута, 5 минут, 15 минут, 60 минут.
3. THE RetryQueue SHALL выполнить не более 4 попыток доставки одного уведомления.
4. IF все 4 попытки доставки завершились неудачей, THEN THE System SHALL записать финальную ошибку в AuditLog со статусом «Не доставлено» и прекратить попытки.
5. THE System SHALL обеспечивать идемпотентность задач: повторное выполнение одной задачи не должно приводить к дублированию отправленных сообщений.
6. THE RetryQueue SHALL хранить состояние задач в PostgreSQL для обеспечения устойчивости к перезапускам сервиса.
7. WHILE система перегружена (очередь содержит более 1000 задач), THE Scheduler SHALL приостановить добавление новых задач опроса и записать предупреждение в лог.

### Requirement 10: База данных и хранение данных

**User Story:** Как разработчик, я хочу, чтобы все данные системы хранились в PostgreSQL с чёткой схемой, чтобы обеспечить надёжность и возможность аналитики.

#### Acceptance Criteria

1. THE System SHALL использовать PostgreSQL версии 14 или выше в качестве основной базы данных.
2. THE System SHALL хранить следующие сущности в отдельных таблицах: `clients` (привязки каналов), `notification_templates` (шаблоны), `notification_log` (история отправок), `scheduled_reminders` (запланированные напоминания), `task_queue` (состояние задач).
3. THE System SHALL применять миграции базы данных через Alembic при каждом запуске или обновлении.
4. THE AuditLog SHALL хранить для каждой отправки: `client_id`, `channel`, `notification_type`, `appointment_id`, `status`, `error_message`, `sent_at`, `delivered_at`.
5. THE System SHALL использовать индексы на полях `client_id`, `appointment_id`, `sent_at` в таблице `notification_log` для обеспечения производительности запросов.
6. WHEN запись в `notification_log` старше 90 дней, THE System SHALL архивировать или удалять её в рамках ежедневной задачи очистки.
7. THE System SHALL использовать пул соединений с базой данных (SQLAlchemy connection pool) с минимальным размером 5 и максимальным 20 соединений.

### Requirement 11: Административная панель

**User Story:** Как администратор, я хочу иметь удобный интерфейс для управления системой, просмотра статистики и настройки шаблонов, чтобы контролировать работу бота без доступа к коду.

#### Acceptance Criteria

1. THE AdminPanel SHALL предоставлять веб-интерфейс, доступный по адресу `/admin`, защищённый HTTP Basic Auth или JWT-токеном с учётными данными из переменных окружения.
2. THE AdminPanel SHALL отображать дашборд со следующими метриками за выбранный период: количество отправленных уведомлений по каналам, процент успешной доставки, количество ошибок по типам.
3. THE AdminPanel SHALL предоставлять интерфейс просмотра и редактирования шаблонов сообщений для каждого типа уведомления и каждого канала.
4. THE AdminPanel SHALL отображать историю отправок с фильтрацией по каналу, типу уведомления, статусу и периоду.
5. THE AdminPanel SHALL предоставлять возможность ручной отправки тестового уведомления на указанный идентификатор клиента.
6. THE AdminPanel SHALL отображать список клиентов с привязанными каналами и возможностью деактивации отдельных каналов.
7. WHEN администратор изменяет шаблон через AdminPanel, THE System SHALL применить изменения для всех последующих уведомлений без перезапуска сервиса.
8. THE AdminPanel SHALL поддерживать управление через Telegram-команды для администратора бота (команды `/stats`, `/template`, `/clients`) как альтернативный интерфейс.

### Requirement 12: Логирование и мониторинг

**User Story:** Как DevOps-инженер, я хочу иметь полное логирование всех операций системы, чтобы быстро диагностировать проблемы в production.

#### Acceptance Criteria

1. THE System SHALL использовать структурированное логирование в формате JSON с полями: `timestamp`, `level`, `service`, `event`, `client_id`, `channel`, `error`.
2. THE System SHALL записывать логи уровня INFO для каждой успешно отправленной нотификации и уровня ERROR для каждой ошибки доставки.
3. THE System SHALL записывать логи уровня WARNING при срабатывании RateLimiter или при недоступности внешнего API.
4. THE System SHALL выводить логи в stdout для совместимости с Docker-окружением и системами сбора логов.
5. THE RateLimiter SHALL ограничивать частоту запросов к каждому внешнему API согласно их документации: Telegram — 30 сообщений/сек, WhatsApp — 80 сообщений/сек, MAX — согласно документации API.
6. THE System SHALL предоставлять эндпоинт `/health` для проверки работоспособности, возвращающий статус подключения к PostgreSQL, Redis и внешним API.
7. THE System SHALL предоставлять эндпоинт `/metrics` в формате Prometheus для интеграции с системами мониторинга.

### Requirement 13: Конфигурация и развёртывание

**User Story:** Как DevOps-инженер, я хочу развернуть систему через Docker Compose с минимальными усилиями, чтобы быстро запустить её в production.

#### Acceptance Criteria

1. THE System SHALL предоставлять `docker-compose.yml`, запускающий все компоненты: FastAPI-приложение, Celery-воркер, PostgreSQL, Redis, AdminPanel.
2. THE System SHALL читать все секреты и конфигурационные параметры исключительно из переменных окружения, перечисленных в файле `.env.example`.
3. THE System SHALL включать следующие обязательные переменные окружения: `YCLIENTS_API_TOKEN`, `YCLIENTS_COMPANY_ID`, `TELEGRAM_BOT_TOKEN`, `WHATSAPP_API_TOKEN`, `WHATSAPP_PHONE_NUMBER_ID`, `MAX_BOT_TOKEN`, `DATABASE_URL`, `REDIS_URL`, `ADMIN_USERNAME`, `ADMIN_PASSWORD`.
4. THE System SHALL предоставлять файл `.env.example` с описанием всех переменных окружения и примерами значений (без реальных секретов).
5. WHEN Docker-контейнер запускается, THE System SHALL выполнить миграции базы данных и проверить доступность всех внешних сервисов перед началом обработки задач.
6. IF обязательная переменная окружения отсутствует при запуске, THEN THE System SHALL завершить работу с кодом выхода 1 и сообщением об ошибке, указывающим имя отсутствующей переменной.
7. THE System SHALL предоставлять `Dockerfile` с многоэтапной сборкой (multi-stage build) для минимизации размера production-образа.
8. THE System SHALL включать `README.md` с инструкцией по развёртыванию, описанием всех переменных окружения и примерами команд.

### Requirement 14: Безопасность

**User Story:** Как владелец бизнеса, я хочу, чтобы система безопасно хранила токены и персональные данные клиентов, чтобы соответствовать требованиям безопасности и законодательству.

#### Acceptance Criteria

1. THE System SHALL хранить все API-токены и пароли исключительно в переменных окружения и никогда не записывать их в логи, базу данных или исходный код.
2. THE WebhookHandler SHALL валидировать подпись входящих webhook-запросов от YClients с использованием секретного ключа из переменной окружения `YCLIENTS_WEBHOOK_SECRET`.
3. THE AdminPanel SHALL использовать HTTPS в production-окружении и отклонять HTTP-запросы с перенаправлением на HTTPS.
4. THE System SHALL применять параметризованные SQL-запросы через SQLAlchemy ORM для предотвращения SQL-инъекций.
5. THE System SHALL ограничивать частоту запросов к эндпоинту `/webhook/yclients` до 100 запросов в минуту с одного IP-адреса.
6. THE System SHALL хранить номера телефонов клиентов в базе данных в хешированном виде (SHA-256) для поиска, сохраняя оригинальный номер только для отправки сообщений.
7. WHEN срок действия JWT-токена AdminPanel истекает, THE AdminPanel SHALL потребовать повторную аутентификацию без сохранения сессии на стороне сервера.
